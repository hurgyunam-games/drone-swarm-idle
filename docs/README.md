# 문서 허브 (docs/)

이 폴더는 **기획(Planning)** 과 **개발 명세(Spec)** 를 분리해 관리하는 문서 허브입니다.
문서의 목적을 분리해 AI/개발자 모두가 같은 맥락으로 작업하도록 하는 것을 목표로 합니다.

---

## 문서 체계

### 1) Planning (왜/무엇)
- 게임 의도, 플레이어 경험 목표, 코어 루프, 밸런스 방향, UX 목표를 정의
- "무엇을 만들지"와 "왜 필요한지"를 설명
- 경로: `planning/`

### 2) Spec (어떻게)
- 구현 가능한 수준의 기술 명세를 정의
- 클래스 다이어그램, 시퀀스 다이어그램, 인터페이스 계약, 상태/규칙, 테스트 기준 포함
- 경로: `spec/`

---

## 권장 작업 순서

1. `planning/`에서 목적과 범위를 확정
2. `spec/`에서 구조/흐름/계약을 구현 가능 수준으로 구체화
3. `spec` 문서를 기준으로 코드 구현
4. 구현 중 변경사항은 `spec` 우선 반영 후 필요 시 `planning` 동기화

---

## 개발 가능 문서 기준 (Spec 최소 세트)

- 요구사항/범위: 목표, 제외 범위(Out of Scope), 완료 조건
- 구조: 클래스 다이어그램(책임, 의존 방향, 데이터 소유권)
- 흐름: 시퀀스 다이어그램(정상 경로 + 실패/예외 분기)
- 계약: 입력/출력 스키마, 이벤트 이름, 에러 규약
- 규칙: 상태 전이, 제약조건, 계산 규칙
- 검증: 테스트 시나리오, 인수 기준(Definition of Done)

---

## 문서 작성 규칙

- 모든 문서는 **Markdown(.md)** 형식으로 작성
- 미확정 사항은 `TODO:` 태그로 명시
- 다이어그램 변경 시 관련 텍스트 명세(규칙/계약/테스트)도 함께 갱신
- AI에게 구현을 요청할 때는 **`spec` 문서를 우선 참조**하고, `planning`은 의도 확인용으로 함께 참조

## 마이그레이션 메모

- 기존 `concept/` 문서는 단계적으로 `planning/concept/`로 이동 권장
- 기존 `mechanics/`, `balance/`, `ui-ux/`는 즉시 삭제하지 않고 유지
- 신규 문서는 `planning/` 또는 `spec/` 하위에 우선 생성
