# 범위 및 분석 규칙

## 0) 폴더별 적용 시점
- `01_core_loop`: 시스템 분석 문서가 2~3개 쌓인 뒤 전체 루프를 요약할 때 사용
- `02_player-experience`: 플레이 로그 2회 이상 쌓인 뒤 감정 패턴을 정리할 때 사용
- `03_systems`: 매 세션 직후 가장 먼저 작성하는 핵심 분석 폴더
- `04_content-structure`: 이벤트/세력/아이템 분류가 필요해질 때 사용
- `05_balance`: 초중후반 체감 차이를 비교할 데이터가 생긴 뒤 사용
- `06_meta`: 바닐라 분석이 안정된 뒤 DLC/모드 차이를 볼 때 사용
- `07_benchmarks`: 다른 게임 비교가 필요해질 때 사용
- `08_outputs`: 매주 1회, 분석 결과를 실험 태스크로 변환할 때 사용

## 1) 분석 범위
- 대상 버전: (예: 최신 안정 버전)
- 포함 콘텐츠: (예: 바닐라 / 특정 DLC 포함)
- 제외 콘텐츠: (예: 모드, 멀티플레이 변형)
- 플레이 난이도/시나리오: (예: Cassandra, Strive to Survive)
- 관찰 단위: (예: 1세션 60~90분)

## 2) 이번 사이클 목표 (2~4주)
- 최우선 분석 시스템 3개:
  - 예) Storyteller
  - 예) Needs/Mood
  - 예) Economy/Trade
- 산출물:
  - 시스템 분석 문서 N개
  - 적용 가능한 기획 원칙 N개
  - 내 프로젝트 태스크 N개

## 3) 기록 규칙
- 사실(관찰)과 해석(가설)을 분리해서 작성한다.
- 한 문서에 하나의 시스템만 다룬다.
- 문장보다 표/불릿으로 짧게 쓴다.
- "재미있다"라고 쓰지 말고, 어떤 선택과 대가가 있었는지 기록한다.
- 플레이 로그에는 사건을 자유롭게 적고, 이후 `선택-대가-감정` 3줄로 가공한다.
- 수치가 없으면 빈칸 대신 강도/빈도 표현(자주, 크게, 즉시, 장기)을 사용한다.

## 4) 평가 질문 (문서마다 체크)
- 이 시스템은 어떤 플레이어 행동을 유도하는가?
- 실패했을 때 복구 경로가 존재하는가?
- 리스크와 보상이 명확하게 연결되는가?
- 초반/중반/후반에서 의미가 어떻게 바뀌는가?
- 내 게임에 가져올 때 어떤 요소를 반드시 단순화해야 하는가?

## 5) 금지 사항
- 위키 문구 복사로 문서를 채우지 않는다.
- 결론 없이 기능 나열만 하지 않는다.
- 내 프로젝트 적용 항목 없이 문서를 종료하지 않는다.

## 6) 주간 리뷰
- 이번 주 핵심 인사이트 3개
- 버릴 아이디어 2개
- 다음 주 실험할 설계 가설 1개
