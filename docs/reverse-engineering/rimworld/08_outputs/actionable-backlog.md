# Actionable Backlog

역기획 결과를 내 게임 실험 태스크로 변환하는 문서.

## 사용 규칙
- 한 태스크는 1~2일 내 검증 가능한 단위로 쪼갠다.
- 반드시 `검증 조건`을 작성한다.
- "좋아 보임"이 아니라 "측정 가능함"으로 종료 조건을 만든다.

## 후보 태스크 (Inbox)
- [ ] (예시) 무드 급락 이벤트 발생 시, 회복 선택지를 2개 이상 제공
- [ ] (예시) 초반 10분 내 강제 의사결정 포인트 3개 설계
- [ ] (예시) 실패 후 복구 루프(자원/시간/리스크) 시뮬레이션

## 이번 주 실행 (This Week)
### Task 1
- 제목: 연속 위기 직후 회복 선택지 2개 제공 테스트
- 출처 문서: `03_systems/storyteller-ai.md`
- 가설: 같은 위기라도 "지금 뭘 할지"를 분명히 고르면 덜 억울하고 더 내가 통제하는 느낌이 든다.
- 구현/실험 방식:
  - 큰 사건이 나오면 잠깐 멈추고 선택 2개를 먼저 적는다.
  - 예: A) 생산 계속하기, B) 회복 먼저 하기.
  - 사건이 끝날 때마다 점수만 적는다: 억울함(1~5), 통제감(1~5).
- 검증 조건:
  - 사건 3번 이상에서 평균 통제감이 3 이상이고, 평균 억울함이 3 이하면 성공.
  - 아니면 선택지 내용을 더 쉬운 형태로 바꿔서 다시 테스트.
- 결과: (다음 플레이 후 작성)

### Task 2
- 제목: 생산 우선 vs 회복 우선 운영 정책 비교
- 출처 문서: `03_systems/needs-mood.md`
- 가설: 당장 손해를 보더라도 회복을 먼저 하면 전체 운영이 더 오래 안정적으로 간다.
- 구현/실험 방식:
  - 비슷한 상황에서 두 방식을 번갈아 해본다.
  - A 방식: 생산 먼저.
  - B 방식: 회복 먼저(치료/휴식/안전한 일).
  - 끝나고 3가지만 비교한다: 수습 시간, 추가 손실, 내 피로감.
- 검증 조건:
  - 회복 먼저 했을 때 추가 손실이 줄고 수습이 안정적이면 성공.
  - 반대로 생산 먼저가 더 잘되면, "언제 회복 먼저 할지" 기준을 다시 정한다.
